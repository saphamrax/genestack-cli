Target hosts

Target hosts Configuration

The Rackspace OpenStack Enterprise installation process requires at least five target hosts to contain the OpenStack environment and supporting infrastructure. Perform the following tasks On each target host:

  • Name the target hosts

  • Install the operating system

  • Configure Secure Shell (SSH) keys

  • Configure the operating system

  • Configure the network

Install the operating system

Install the Ubuntu 24.04 LTS Server 64-bit, operating system on the target host with at least one network interface configured to access the Internet or suitable local repositories.

The controller are required to be bootstrapped with the base OS in order to install Rackspace OpenStack. Subsequent hosts are bootstraped with the OpenStack Baremetal Service, Ironic. This enables a shared control plane for OpenStack guests and Baremetal Services.

On target hosts without local (console) access, we recommend adding the Secure Shell (SSH) server packages to the installation.

Configure Secure Shell (SSH) keys

Ansible uses Secure Shell (SSH) to connect the deployment and target hosts:

ssh-keygen -b 4096 -t rsa -C "Ansible Deployment Key"

cat ~/.ssh/id_rsa.pub >> /root/.ssh/authorized_keys
chmod 0600 /root/.ssh/authorized_keys

  • Copy the contents of the public key file on the deployment host to the /root/.ssh/authorized_keys on each target host.

  • Test public key authentication from the deployment host to each target host. SSH should provide a shell without asking for a password.

NOTE In some cases, the root access is not allowed, please enable the SSH key access via

echo "PermitRootLogin without-password" >> /etc/ssh/sshd_config
systemctl restart sshd

Configure the operating system

Check the kernel version, install additional software packages, and configure the network time protocol (NTP).

NTP is no longer required. The hardening role installs and configures chrony to provide network time services on all deployments by default.

Any required host OS configuration and packages to install OpenStack Enterprise is automated with the host-setup.yml playbook. Before installing any component please upgrade the host OS to install all latest patches.

Once the host setup and kubernetes components are installed the host should be rebooted once to use the required kernel and configurations.

!!! Note The following example of the metallb configuration splits a "/24" network into two "/25" subnets and announce different IP ranges for external vs. internal (primary metallb pool) services. The physical interface is typically br-host for external services, as the default gateway is configured on it, the internal services can be layered on the same interface or moved to non routed interfaces such as br-mgmt

cat << EOT > /etc/genestack/manifests/metallb/metallb-openstack-service-lb.yml
---
apiVersion: metallb.io/v1beta1
kind: IPAddressPool
metadata: 
  name: gateway-api-external 
  namespace: metallb-system 
spec:
  addresses: 
  - 10.230.0.10/32
  autoAssign: false
---
apiVersion: metallb.io/v1beta1
kind: L2Advertisement 
metadata:
  name: openstack-external-advertisement 
  namespace: metallb-system 
spec:
  ipAddressPools: 
  - gateway-api-external 
  nodeSelectors: # Optional block to limit nodes for a given advertisement 
  - matchLabels: 
      node-role.kubernetes.io/worker: worker 
  interfaces: # Optional block to limit ifaces used to advertise VIPs
  - br-host
---
apiVersion: metallb.io/v1beta1
kind: IPAddressPool 
metadata: 
  name: primary 
  namespace: metallb-system 
spec:
    addresses: 
    - 10.230.0.128/25
    autoAssign: false
---
apiVersion: metallb.io/v1beta1
kind: L2Advertisement 
metadata: 
  name: openstack-internal-advertisement 
  namespace: metallb-system 
spec:
  ipAddressPools: 
  - primary 
  nodeSelectors: # Optional block to limit nodes for a given advertisement - 
  -  matchLabels: 
      node-role.kubernetes.io/worker: worker 
  interfaces: # Optional block to limit ifaces used to advertise VIPs 
  - br-host
EOT

S3 Backend configuration Example

!!! Note The S3 buckets must be created ahead of using them with Loki. The aws cli can be used to create buckets after installation and configuration.

It is generally adviseable to setup S3 credentials and buckets per regions/openstack deployment.
If the credentials are shared across regions, ensure that each region is isolated in their respective buckets.

```bash
   pip install --user awscli

   cat << EOT > ~/.aws/config
   [default]
   region = default
   EOT

   cat << EOT > ~/.aws/credentials
   [default]
   aws_access_key_id = <access key>
   aws_secret_access_key = <secret key>
   EOT
```

Once configured check the access with

```bash
   aws s3 ls
```

and create the buckets

```bash
   alias aws='aws --endpoint-url=https://<url>'

   aws s3api create-bucket --bucket regionone-loki-admin
   aws s3api create-bucket --bucket regionone-loki-chunks
   aws s3api create-bucket --bucket regionone-loki-ruler
```

OVN Install and Configure

OVN Annotations

key
 type
value
 notes
ovn.openstack.org/int_bridge str br-int The name of the integration bridge that will be used.
ovn.openstack.org/bridges str br-ex Comma separated list of bridges that will be created and plugged into OVS for a given node.
ovn.openstack.org/ports str br-ex:bond1 Comma separated list of bridge mappings. Maps values from the bridges annotation to physical devices on a given node.
ovn.openstack.org/mappings str physnet1:br-ex Comma separated list of neutron mappings. Maps a value that will be used in neutron to a value found in the ports annotation. Every provider network name listed in this annotation will have a unique mac address generated per-host.
ovn.openstack.org/availability_zones str az1 Colon separated list of Availability Zones a given node will serve.
ovn.openstack.org/gateway str enabled If set to enabled, the node will be marked as a gateway.

Configure external and internal QC Neutron networks

Neutron is preconfigured with a global physnet MTU of 9000 and all networks will inherit it. The DHCP service in neutron will pass along a maximum MTU (1500 for VxLAN) for each port. In some circumstances, it is desired to lower the MTU to prevent fragmentation issues, in such cases the --mtu 1500 option can be added to the openstack network create command. A standard MTU like 1500 is generally advisable for all Internet facing networks like gateway network.

  • Create external (Internet/Public facing) provider network (VLAN only)
   openstack network create GATEWAY_NET --mtu 1500 --external --provider-physical-network \
     physnet1 --share --provider-network-type vlan --provider-segment <VLAN ID>

   openstack subnet create \
     --subnet-range 10.241.0.0/22 --gateway 10.241.0.1 \
     --dns-nameserver <DNS1> --dns-nameserver <DNS2> \
     --network GATEWAY_NET --allocation-pool start=10.241.0.10,end=10.241.3.254 GATEWAY_SUBNET_1
  • Create a external router for RPC QC Testing
   openstack router create --ha RPC_GATEWAY_NET_QC
   openstack router set --enable-snat --external-gateway GATEWAY_NET RPC_GATEWAY_NET_QC
  • Create RPC QC Test Network (VXLAN)
   openstack network create RPC_QC_NET \
     --provider-network-type vxlan

   openstack subnet create \
     --subnet-range 192.168.0.0/24 --gateway 192.168.0.1 \
     --dns-nameserver 1.1.1.1 \
     --network RPC_QC_NET --allocation-pool start=192.168.0.10,end=192.168.0.254 RPC_QC_SUBNET_1
  • Assign RPC QC Network to the Neutron Router
   openstack router add subnet  RPC_GATEWAY_NET_QC RPC_QC_SUBNET_1
